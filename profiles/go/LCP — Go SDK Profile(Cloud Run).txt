0. PURPOSE
============================================================

- Translate LCP v1.0.0 into concrete Go SDK behavior for Cloud Run workloads
- Define normative integration rules for in-memory and storage-backed cache modes
- Preserve all parent protocol invariants without drift

This profile is subordinate to LCP core protocol.

============================================================

- ReadThroughCacheEngine usage in Go services
- Deterministic keying and parity validation in Go

3. LCP SECTION MAPPING (GO PROFILE)
============================================================

I. CORE INVARIANTS

- MUST execute read-through only
- MUST keep API as source of truth
- MUST preserve parity fields and deterministic identity

II. NO-BACKFLOW

- Cache results MUST NOT alter business/protocol decisions
- Background refresh callback MUST remain advisory-only

III. CACHE KEY

- MUST build key from namespace + operationId + canonical payload + schemaVersion + specChecksum + userScope
- MUST exclude trace_id/request_id/action_id from key payload material
- MUST use H57 for hashing and B57 canonical byte material handling
- ID57/R57/S57 terms MUST be used where identifier/random/derivation rules apply

IV. CACHE ENTRY

- Metadata fields REQUIRED:
  source, created_at, expires_at, ttl_ms, schema_version, data_version,
  spec_checksum, cache_namespace, compressed
- Metadata parity MUST match configured contract

V. EXECUTION MODEL

- Ordered path MUST be: memory -> optional persistent store -> API
- Missing TTL MUST bypass cache write and return API response
- Stale fallback MUST be gated by allowStaleOnExpired
- Foreground path MUST keep one request -> one API fetch execution path

VI. STORAGE MODEL

- Memory store REQUIRED
- Persistent store OPTIONAL
- Supported modes: auto | memory-only | dual

VII. TTL MODEL

- TTL MUST come from upstream result only (ttl_ms-compatible metadata)
- Local TTL synthesis for writes MUST NOT be introduced

VIII. SWR MODEL

- Stale return MAY be enabled
- Refresh MUST be caller-managed
- Engine MUST NOT emit implicit refresh RPC

IX. SINGLE-FLIGHT

- Single-flight is OPTIONAL
- It is enabled only when CacheSingleFlight is supplied
- Scope MUST be cache_key

X. STATE ALIGNMENT

- When state context exists, stale entries violating state_version ordering MUST be bypassed
- Resume/state tracking MUST remain advisory

XI. TRACE RULES

- Trace context MAY be carried for request context
- Trace fields MUST NOT be persisted in cache payload data
- Trace fields MUST NOT be included in key material

XII. VALIDATION

- Deterministic serialization and parity checks MUST be enforced
- Sensitive-field and trace-field rejection policy MUST be enforced
- Default validation failure handling MUST be BYPASS_TO_API
- HARD_FAIL MUST be limited to fail-closed or invariant-critical policy

XIII. FAILURE MODEL

- Decision classes:
  BYPASS_TO_API, RETURN_STALE_AND_REFRESH, HARD_FAIL
- storage/lookup/serialization issues SHOULD resolve as BYPASS_TO_API or RETURN_STALE_AND_REFRESH
- invariant-critical/fail-closed policy MUST resolve as HARD_FAIL

XIV. INVALID CONDITIONS

- Any semantic mutation or non-deterministic identity behavior is invalid

XV. GUARANTEES

- Deterministic read-through behavior
- Replay-safe keying and parity-safe caching
- Optional single-flight and optional SWR stale fallback
- No-backflow behavior for execution/state/observability pathways

4. CLOUD RUN INTEGRATION CONTRACT
============================================================

REQUEST (Execute):

- keyInput:
  namespace, operationId, payload, schemaVersion, specChecksum, userScope
- H57HashFn
- fetchFromAPI -> { data, ttl_ms?, headers? }
- allowStaleOnExpired?
- resumeState?
- trace?
- onBackgroundRefresh?

RESPONSE:

- cacheKey
- source: API | CACHE
- data
- stale: bool

Behavioral constraints:

- fetchFromAPI is the authoritative upstream fetcher
- cache writes MUST occur only after successful parity/invariant validation
- missing TTL responses MUST return source=API, stale=false

============================================================
5. CLOUD RUN STORAGE PATTERNS
============================================================

PATTERN A: IN-MEMORY + IN-MEMORY PERSISTENT STORE

- SDK persistent layer MAY use NewInMemoryPersistentStore for process-local persistence

PATTERN B: SQLITE + SQLITE PERSISTENT STORE

- SDK sqlite store path SHOULD be explicitly configured (example: /tmp/lcp/lcp_cache.db)
- storage backend selection SHOULD use LCP_STORAGE_BACKEND=sqlite

PATTERN C: CLOUD-STORAGE + DIRECT CLOUD STORAGE PERSISTENT STORE

- storage backend selection SHOULD use LCP_STORAGE_BACKEND=cloud-storage
- storage target SHOULD use LCP_STORAGE_GCS_URI (example: gs://bucket/lcp)
- runtime identity or service-account credentials MUST have object read/write access

============================================================
6. CLOUD RUN OPERATIONAL RULES
============================================================

- mounted paths MUST avoid disallowed mount targets (/dev, /proc, /sys)
- cloud-storage backend MUST use valid gs:// URI semantics
- sqlite backend path SHOULD be writable by service runtime
- startup behavior MUST tolerate backend init failures with safe cache bypass behavior

============================================================
7. CONFORMANCE CHECKLIST (GO CLOUD RUN)
============================================================

A Go integration is conformant when all are true:

- Uses ReadThroughCacheEngine read-through path
- Uses deterministic key material and hash
- Preserves configured parity fields
- Does not persist sensitive or trace fields in cache payload data
- Bypasses cache write when TTL is missing
- Treats SWR refresh callback as caller-managed
- Does not assume single-flight unless configured
- Preserves no-backflow invariants under all cache outcomes
- Uses one of the approved Cloud Run persistence patterns

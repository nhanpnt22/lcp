4. STAGE TASK CONTRACTS (UNIVERSAL)
============================================================

4.1 TEST DESIGN

- define test-first contracts (unit, integration, acceptance)
- include deterministic expected outputs

------------------------------------------------------------

4.2 STORY REGISTRATION

- persist story in state store
- assign deterministic identity

RULE:
→ no story → no task split

------------------------------------------------------------

4.3 PLANNING

- decompose story into roles and stages
- map acceptance criteria to tasks

------------------------------------------------------------

4.4 TASK REGISTRATION

- persist ALL tasks BEFORE execution
- include identity, stage, dependencies, I/O contracts

RULE:
→ partial registration = HARD FAIL

------------------------------------------------------------

4.5 GRAPH BUILD

- construct DAG (nodes + edges)
- enforce acyclic structure
- validate identity consistency

------------------------------------------------------------

4.6 KNOWLEDGE RESOLUTION

- resolve dependencies (libraries, APIs, configs)
- validate compatibility and security

RULE:
→ BUILD cannot start without this

------------------------------------------------------------

4.7 KNOWLEDGE UPDATE

- persist validated knowledge into state store
- ensure versioned traceability

------------------------------------------------------------

4.8 CONTEXT INDEX UPDATE

- update retrieval/index system (RAG or equivalent)
- source ONLY from validated internal data

------------------------------------------------------------

4.9 CI VALIDATION

- validate structure, complexity, policies
- block execution on failure

RULE:
→ NO CI PASS → NO BUILD

------------------------------------------------------------

4.10 BUILD

- implement deterministic logic
- avoid randomness or hidden state

------------------------------------------------------------

4.11 QC / QA

- verify correctness and contract integrity
- ensure deterministic outputs

------------------------------------------------------------

4.12 EXECUTION

- run sequentially
- persist execution evidence
- stop on failure

------------------------------------------------------------

4.13 DRIFT DETECTION

- compare expected vs actual state
- generate remediation tasks if needed

RULE:
→ NO DRIFT DETECTION → INVALID EXECUTION

------------------------------------------------------------

4.14 UAT

- validate business acceptance deterministically

------------------------------------------------------------

4.15 VALIDATION (COREX-equivalent)

- validate checksum, replay, dependencies
- ensure deterministic correctness

------------------------------------------------------------

4.16 CACHE VALIDATION

- verify identity consistency
- invalidate stale cache deterministically

------------------------------------------------------------

4.17 RELEASE

- allow release ONLY if all gates pass
- include full execution evidence

7. STATUS MODEL (UNIVERSAL)
============================================================

Derived ONLY from execution evidence:

- PENDING
- READY
- RUNNING
- DONE (exactly ONE success)
- FAILED (all failed)
- BLOCKED (dependency failure)

RULE:

→ no manual updates  
→ status = pure function(execution_history + dependencies)

9. DEPENDENCY LAW
